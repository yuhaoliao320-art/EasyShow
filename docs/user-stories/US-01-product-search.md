---

## 🎯 User Story

**As a** 前台瀏覽訪客（一般消費者／設計師／工班）
**I want** 在網站上輸入關鍵字搜尋產品名稱
**So that** 我可以快速找到想看的特定材料，不用逐層翻分類

---

## 📐 技術脈絡

- 後端 Supabase 已有 `searchProducts()` API（使用 PostgreSQL `ILIKE` 模糊搜尋，僅回傳已上架產品，上限 50 筆）
- 目前前台**沒有搜尋框 UI**，使用者只能透過左側分類樹或首頁分類連結瀏覽
- 需新增前端搜尋 UI 元件並串接既有 API

---

## ✅ 驗收條件（Acceptance Criteria）

### AC-01：全域搜尋框顯示
**Given** 我在前台任何頁面（首頁、分類頁、產品詳情頁等）
**When** 我檢視頁面頂部 header 區域
**Then** 我看到一個搜尋輸入框（帶放大鏡圖示），placeholder 顯示「搜尋產品名稱…」

### AC-02：輸入關鍵字觸發搜尋
**Given** 我在 header 的搜尋框中輸入了關鍵字（例如「磁磚」）
**When** 我按下 Enter 鍵或點擊搜尋圖示
**Then** 系統導航至 `/search?q=磁磚` 搜尋結果頁面

### AC-03：搜尋結果顯示
**Given** 我在搜尋結果頁面
**When** 系統完成搜尋
**Then** 頁面以產品卡片網格顯示所有符合的產品，每張卡片包含：產品名稱、第一張縮圖、所屬分類名稱

### AC-04：無搜尋結果提示
**Given** 我搜尋了一個不存在的關鍵字（例如「zzzxxx」）
**When** 系統回傳空結果
**Then** 頁面顯示「找不到符合「zzzxxx」的產品」並提供建議：
- 檢查關鍵字是否有錯字
- 嘗試使用更通用的詞彙
- 顯示「瀏覽全部分類」的連結

### AC-05：搜尋結果數量標示
**Given** 搜尋完成
**When** 結果頁面載入
**Then** 頁面頂部顯示「搜尋「磁磚」共找到 12 項產品」

### AC-06：輸入即時清除
**Given** 我在搜尋框中輸入了文字
**When** 我點擊搜尋框內的清除按鈕（✕）
**Then** 搜尋框文字被清空，游標保持在搜尋框內

## ⚠️ 邊界案例（Edge Cases）

| 編號 | 情境 | 預期行為 |
|------|------|---------|
| EC-01 | 搜尋空字串或只輸入空白 | 不觸發搜尋，提示「請輸入關鍵字」 |
| EC-02 | 關鍵字前後有空白 | 自動 trim 後再搜尋 |
| EC-03 | 關鍵字包含特殊字元（`!@#$%^&*()`） | 正常執行 ILIKE 搜尋，無需特殊處理 |
| EC-04 | 搜尋結果超過 50 筆 | 顯示「前 50 筆符合結果」，提示精確關鍵字 |
| EC-05 | 快速連續輸入（debounce） | 不需即時搜尋，僅在按下 Enter 或點擊按鈕時觸發 |
| EC-06 | 網路連線中斷 | 顯示「網路異常，請稍後再試」與重試按鈕 |
| EC-07 | 關鍵字包含 HTML 標籤（如 `<script>`） | 將使用者輸入進行 HTML escape 後顯示在頁面上，防止 XSS |

## ⚡ 非功能性需求（NFR）

| 類別 | 需求描述 | 驗收標準 |
|------|---------|---------|
| 效能 | 搜尋 API 回應時間 | 平均 < 500ms，P99 < 2s（產品數 < 5000 筆） |
| 可用性 | 搜尋框在所有瀏覽器（Chrome / Safari / Firefox）上正常顯示 | 按鈕與 icon 位置一致 |
| 可用性 | 支援 RWD | 手機版搜尋框可展開為全寬 |
| 可近用性 | 支援鍵盤操作 | Tab 可聚焦到搜尋框，Enter 觸發搜尋 |
| SEO | 搜尋結果頁面 | 使用 `<noindex>` 避免被索引 |

## 📝 實作建議

1. **搜尋框元件化**：建立獨立的 `SearchBar.tsx` 元件，可複用於 header 與其他位置
2. **路由設計**：`/search?q=關鍵字`，使用 `useSearchParams` 讀取 query
3. **API 擴充建議**：未來可考慮加入 PostgreSQL `tsvector` 全文檢索取代 `ILIKE`，以支援更精確的搜尋
4. **快取策略**：相同關鍵字的搜尋結果可在 session 內快取 5 分鐘，減少重複請求
